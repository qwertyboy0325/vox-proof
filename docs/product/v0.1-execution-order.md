```text

A line may contain a canonical term alone. Every non-canonical source form must use either `alias:` or `error:`. The sealed FLEURS probe predates native canonical-only support and used self-referential `alias:<canonical>` entries solely for then-required syntactic compliance; that remains historical sealed-study provenance, not the current required input form. This provisional session format is not automatic learning, correction memory, a Language Pack, persistence format, or product-ready glossary system.

## Pain-Point MVP Loop

Decided 2026-07-10. The first testable MVP is not a general subtitle QA tool. It targets one narrow pain, held as a targeted hypothesis, not an established fact (reported by Lead A; see `docs/discussion/2026-07-09.md` and `product/hypotheses.md`):

> ASR subtitle drafts are reported to misrecognize or inconsistently render proper nouns, place names, brand names, named entities, and mixed Chinese-English terms; these errors are reported to be costly because editors must notice them manually and often repeat the same corrections across content.

Instrumented real-material runs, their human-confirmed comparisons, and the eventual Lead A session test this hypothesis.

The minimum pain-point MVP loop:

```text
raw ASR SRT
-> session terms / glossary / observed error-form hints
-> candidate evidence engine
-> ReviewCases
-> human decisions
-> reviewed SRT
-> decision log
-> correction profile / session summary
-> local evaluation inputs / run metrics
```

The candidate evidence engine raises review-worthy candidates with evidence. It does not own truth and does not correct subtitles. Reviewed output derives only from accepted human decisions (MD-003).

Correction profile constraints:

- derived solely from the session's decision log at export time;
- human-readable summary output only, with no machine re-import format;
- a future reusable-asset seed observation, not a Domain Collection, Language Pack, or Knowledge Pack. Any re-import capability belongs no earlier than v0.3 and requires its own future decision.

Implemented session-summary metrics are limited to run identity and display paths, transcript and session-term counts, candidate counts by detection kind and detector provenance, effective decision counts plus total ledger events, accepted replacement and affected-segment counts after successful materialization, and local session timing. Start and end use Unix epoch milliseconds; elapsed duration uses the monotonic local session timer and is also rendered as `HH:MM:SS.mmm`. Summary generation has reviewed-SRT materialization and decision-log rendering as preconditions; the collector does not manufacture success facts for operations it did not perform. These are run facts, not precision, recall, correctness, or workflow-value claims.

Output writes remain a bounded sequential filesystem operation, not a transaction. The CLI reports overall success only after all three files are written. If a later write fails, it returns an incomplete-output error without printing success messages, but an earlier file may remain on disk.

Calibration observation from the 2026-07-14 synthetic smoke test: canonical-term authoring and anchor selection must account for adjacent-script spacing. Exact accepted-span replacement can produce text such as `去SHIBUYA SKY` when the source has no space before the matched alias. This observation does not change MD-003 and does not authorize automatic whitespace insertion, script-boundary formatting, post-processing normalization, or edits outside the accepted span. Any replacement-boundary semantics require a separate future decision informed by real material.

## Embedded Calibration Loop

Implemented for strict skeleton-compatible inputs:

```text
vox-proof compare <raw-input.srt> <final-input.srt> <comparison-report.json>
```

This produces provisional schema `voxproof-calibration-comparison-v0` under compatibility policy `identical-cue-count-index-and-timing-v0`. It records per-cue `unchanged` and `text_changed` facts only. It does not join detector misses, classify accuracy/formatting/editorial changes, compute metrics, or align mismatched segmentation or timing.

Implemented local correspondence inventory for strict skeleton-compatible inputs:

```text
vox-proof evaluate <raw-input.srt> <final-input.srt> <session-terms.txt> <evaluation-report.json>
```

This produces provisional schema `voxproof-calibration-correspondence-v0` under the same compatibility policy, authorized by MD-006. It deterministically inventories cue-local raw-to-final edits, canonical ReviewCases generated from raw, and neutral correspondence facts. It does not compute precision, recall, effectiveness metrics, certify comparison-final provenance, or establish product validation.

The completed mechanism probes do not constitute the first measured error-distribution study. A future qualifying human-confirmed real-material run creates that distribution; the project does not require it before completing the bounded MVP.

For one real, self-owned or authorized ASR draft plus its human-corrected final version, the calibration loop is:

```text
raw ASR SRT
+ effective session terms / observed forms / detector configuration
+ review cases and decision log
+ human-corrected final SRT
-> local diff / evaluation report
-> measured error-form distribution and detector misses
-> bounded detector or threshold adjustment
```

The run should capture, as locally inspectable session output where the implementation supports it:

- effective detector identity, version, and configuration;
- number of raised review cases;
- accept, reject, defer, and needs-manual-correction counts;
- accepted-correction and rejected-candidate rates;
- per-case or session handling time when practical;
- comparison against human-confirmed final corrections when that final material exists.

The human-confirmed comparison answers:

- how many real corrections exist;
- what fraction are enumerable term / proper noun / mixed-language errors;
- whether error forms are stable or reusable across content;
- what deterministic alias / observed-form / phonetic evidence plausibly catches;
- what residual class remains after the deterministic rungs.

Evidence rules:

- Validation evidence must come from real ASR output and real human correction. Synthetic or model-generated material is smoke-test material only, never validation evidence.
- Model pre-labeling of validation material is permitted only as a drafting aid. Every item in the recorded ground truth requires individual human confirmation, and model assistance is noted in provenance.
- For v0.1 establishment, the project owner may act as the real human reviewer only under the qualifying procedure in MD-007 D8. A real human reviewer means an actual listening and adjudication act. Community or reference text may support localization but may not replace human listening or independent human judgment.
- External target-cohort or facilitated-user testing remains deferred to v0.2.

## Detector Evidence Ladder

Amended 2026-07-14. The evidence path is a delivery and calibration ladder. The ladder orders the evidence sources declared in `product/v0.1.md`; it does not add scope. Rungs 1 and 2 are independent of each other and may be implemented with conservative provisional behavior before the first measured error distribution so the usable MVP and its validation loop can be completed together. 0. Exact canonical alias matching. Implemented with provisional, user-supplied session-term input.
1. Observed-error-form mapping (`observed_error_form -> canonical_term`). Implemented as exact, case-sensitive deterministic matching with distinct typed evidence and detector provenance. It shares the accepted `GlossaryAliasMatch` detection kind with ordinary aliases; summary calibration distinguishes the paths by detector ID. Actual observed forms must come from user-supplied or human-confirmed material rather than invented entries presented as evidence.
2. Bounded ASCII-Latin phonetic similarity against session terms. Implemented for the provisional v0.1.0 producer under MD-005; MD-004 owns the identity and human-authority boundary. Independent of rung 1. CJK/pinyin, broader phonetic eligibility, and future detector expansion remain future work and must remain evidence producers rather than correction authority; provisional configuration must stay explicit, inspectable, and versioned.
3. Local open-source non-LLM evidence model, acting only as an evidence producer behind the same CandidateSpan / ReviewCase boundary as deterministic detectors. Allowed in technical posture; not yet justified as an implementation dependency. Gates are defined in the model-use boundary below.
4. LLM-like candidate proposer: outside v0.1 entirely. Not on the v0.1 path; would require its own scope decision.

Mixed-language signal (a declared v0.1 evidence source) is an orthogonal suspicion signal, not a matcher: it can flag a span without proposing an alternative. It remains lower priority than completing and instrumenting the term-focused loop; real runs determine whether it is useful.

Rejected model classes for this wedge, recorded 2026-07-10: NER / generic token classification, and semantic embedding similarity. Rationale: the ASR corruption channel is phonetic, not semantic, and misrecognized entities often no longer look like entities. Reopen only if real-material data contradicts this.

## Model-Use Boundary: Three Zones

Decided 2026-07-10. The underlying scope non-goals are owned by `product/v0.1.md`; this section records the operational boundary for execution.

Zone A: development-time model assistance. Allowed. Coding assistance, tests, docs, refactors, design alternatives, review notes. Creates no product or runtime dependency.

Zone B: model-assisted data artifacts. Allowed with human review and provenance: draft glossary entries, proposed observed-error-form mappings, synthetic smoke-test fixtures, pre-labeling material for human review. These artifacts are never validation evidence by themselves; the human-confirmation rule in the Embedded Calibration Loop section applies.

Zone C: product-runtime model dependency. For v0.1: cloud-side model calls are rejected; sending user subtitle material to hosted model services is rejected; local LLM runtime dependency is rejected; model output is never correction authority; rules-only operation must remain possible.

A future local open-source non-LLM evidence model (ladder rung 3) may be considered only as an evidence producer behind the CandidateSpan / ReviewCase boundary. No model runtime, weights distribution, or inference dependency is introduced until both:

1. a real-material diff study identifies a residual error class that deterministic detectors measurably miss; and
2. an accepted Material Decision defines runtime, packaging, privacy, evaluation, threshold governance, and fallback boundaries.
