# v0.1 Execution Order

Goal: complete the minimum closed loop and let a real user (Lead A, see `docs/discussion/2026-07-09.md`) experience the product vision in a facilitated session.

The loop that must close:

```text
existing SRT
-> parse / validate
-> glossary ReviewCases
-> human accept / reject / defer / needs-manual-correction
-> decision log
-> reviewed SRT derivation
```

This document records order, not scope. If it conflicts with canonical scope or governance documents, those documents win. Reorder only with an explicit note here.

## Track 0: Real Material (non-code, start immediately)

Ask Lead A for one episode's ASR draft plus final delivered subtitles, with explicit authorization. This has lead time and blocks matching-semantics and detector decisions, so it starts before and runs alongside all coding work.

Track 0 is not a code freeze.

Lead A material acquisition runs in parallel with Track 1. Real material is required before making detector and evaluation decisions that depend on observed error distribution, such as matching semantics v0, regression cases, and any phonetic-detector work. It does not block closed-loop mechanics that are already governed by accepted decisions, such as decision recording, conservative materialization semantics, reviewed-output derivation, session decision-log output, and a facilitated CLI review flow.

If Lead A material is delayed, continue Track 1 with synthetic or manually constructed examples only as smoke tests. Those examples must not be treated as product validation evidence.

## Track 1: Closed Loop (code, in order)

1. CorrectionDecision slice.
   Decision actions and ledger-event semantics are already accepted in MD-002. No new governance needed. Implement the decision events and in-memory fold.

2. MD-003: minimal materialization semantics.
   The materialization gate in `docs/architecture/pending-data-contracts.md` requires this decision before implementation. Keep it narrow: single-anchor edits, no overlapping accepted edits, revision mismatch means not applicable, any conflict refuses output with an explicit report.

3. Reviewed output derivation.
   Source transcript plus accepted decisions derives a reviewed SRT. Deterministic; conservative; never partial output claimed as success.

4. Decision log output.
   Write review-ledger events to a file at the end of a session. This is session output, not a persistence architecture.

5. Minimal CLI review flow.
   Case-by-case accept / reject / defer in the terminal. Enough for a facilitated session; not a product UI.

## Insert When Track 0 Material Arrives

- Matching semantics v0, informed by the observed error distribution.
- Eval harness and first regression cases from the authorized material.
- Only then evaluate whether a phonetic detector is worth building.

Until this material arrives, do not replace evidence with speculation: continue closed-loop mechanics, but defer data-informed detector semantics.

## Explicit Non-Goals For This Push

- No GUI (the first user test is a facilitated concierge session, not self-serve).
- No phonetic detector before real error data exists.
- No persistence architecture beyond session output files.
- No audio, ASR, LLM, MCP, mobile, or hardware work.

## Validation Criteria For The Lead A Session

Measure three things:

1. True positives: how many real errors in one episode the glossary detector surfaces, using a channel glossary built with Lead A.
2. Time: Lead A's normal correction time for one episode versus the time through this flow.
3. Vision reaction: whether the decision log and reviewed SRT read as valuable to Lead A, or as noise. This tests the flywheel hypothesis, not just the detector.

If the detector finds nearly nothing on real material, that is a result, not a failure of the session: it redirects detector priorities before more detector code is written.
