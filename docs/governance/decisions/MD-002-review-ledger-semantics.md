# MD-002: ReviewCase and Review Ledger Semantics

Status: accepted

Date: 2026-07-08

Decision authority: Ezra

## Context

VoxProof's long-term durable asset is not a detector finding.

The durable asset is an append-only review ledger over review cases, source identity, human decisions, corrections, missed detections, and regression assets.

The current code has a `ReviewCase` wrapper around `CandidateSpan`. That is acceptable for the current detector-only slice, but it must not become the permanent semantic center.

If the ledger is keyed directly to detector output, VoxProof can only learn from what detectors already saw.

Human reviewers must eventually be able to raise review cases when the system misses an issue.

## Decision

`ReviewCase` is the unit of review.

`CandidateSpan` is not the root object of the review world. It is one possible origin of a review case.

Decisions bind to review case identity and observed transcript revision identity, not directly to detector identity.

```text
ReviewCase is the unit of review.
ReviewCase receives decisions.
Review ledger records events about ReviewCases.
CandidateSpan is one possible ReviewCase origin.
```

## ReviewCase origin

A review case has exactly one origin.

The origin set is closed and may grow only by explicit Material Decision.

Initial origin:

```text
DetectorRaised
```

Future origin:

```text
HumanRaised
```

Conceptual model:

```text
ReviewCase
  id: ReviewCaseId
  origin: ReviewCaseOrigin

ReviewCaseOrigin
  DetectorRaised(CandidateSpan)
  HumanRaised(ManualFinding)
```

This is conceptual. The current implementation may continue to support only detector-raised cases until the review ledger is ready.

## Source revision

A review case's source revision is the revision of its origin anchor.

There must be one semantic source of truth for a case's source revision.

If an implementation stores the revision separately for performance or convenience, it must be derived from and consistent with the origin anchor. The semantic source of truth remains the origin anchor.

## DetectorRaised lifecycle

A detector-raised case is a run-scoped derivation.

It is reproducible from:

```text
same transcript revision
same analysis snapshot
same detector identity/version/configuration
same source anchor semantics
```

A detector-raised case may expose detector provenance, `CandidateKey`, evidence, and alternatives through its origin.

But the review ledger must not use detector identity as the primary target of decisions.

## HumanRaised lifecycle

A human-raised case is a revision-scoped human event.

It is not regenerated by analysis reruns.

Rerunning analysis must neither regenerate, duplicate, nor invalidate existing human-raised cases.

A human-raised case exists because a reviewer observed something against a transcript revision.

## Human testimony

Human observations are testimony.

Testimony is:

* recorded
* attributable when attribution exists
* non-recomputable
* allowed to be free text
* capable of being wrong

Human testimony is not machine evidence.

Human testimony is not a decision.

Do not model human testimony as `Evidence`.

Do not model human testimony as a detector output.

The prohibition on generic `reason: String` evidence applies to machine evidence. It does not prohibit free-text human testimony.

## ManualFinding v0

A future v0 text-only human-raised case should require:

* a non-empty `SourceAnchor`
* at least one of:

  * a non-empty reviewer note
  * a proposed correction

Conceptual shape:

```text
ManualFinding
  anchor: SourceAnchor
  note: Option<NonEmptyText>
  proposed_correction: Option<String>
```

A human-raised case with only an anchor and no content is not useful enough as a durable review asset.

## Raising is not deciding

Raising a case and deciding a case are separate ledger events.

This is true even if the same human performs both in one user gesture.

Correct flow:

```text
raise review case
-> record case-raised event
-> record later decision event
```

Wrong flow:

```text
human raised case
= automatically accepted correction
```

Case creation must never silently accept, apply, or materialize a correction.

## Review ledger

The durable structure is a review ledger, not merely a decision list.

The review ledger may eventually contain events such as:

```text
CaseRaised
DecisionRecorded
CaseWithdrawn
```

Future event types may be added only by explicit decision.

For detector-raised cases, case creation can be derived from analysis output.

For human-raised cases, case raising is itself a durable review-ledger event.

Persisting only later decisions is insufficient for human-raised cases because it loses recall provenance.

## Decision binding

Decision records must bind to:

```text
ReviewCaseId
observed TranscriptRevisionId
```

Decision records must not bind directly to:

```text
CandidateKey
detector provenance
CandidateSpan alone
```

A detector-raised review case may still expose `CandidateKey` through its origin, but the ledger target is the review case.

This rule prevents future human-raised cases from becoming second-class or requiring a schema break.

## Review status

Review status is derived from the append-only ledger.

`ReviewCase` must not store mutable review status.

v0 fold rule:

```text
For a given ReviewCase, later decision events supersede earlier decision events
when deriving current review status.
```

This is last-decision-wins for the current effective status.

Earlier events remain in the ledger and are not deleted.

If a future product need requires a different fold rule, that requires a new Material Decision.

## Withdrawal

Humans can misclick, misjudge, or create a mistaken human-raised case.

Mistaken human-raised cases must be corrected by appending a withdrawal event.

They must not be deleted.

Withdrawal event shape is deferred.

## Duplicate human-raised cases

Duplicate human-raised cases are allowed in v0.

Two reviewers, or the same reviewer twice, may raise the same issue at the same or overlapping anchor.

These are separate review cases and separate ledger events.

Deduplication, merging, grouping, or ranking is a later presentation/reconciliation concern.

## Decision actions

The first detector-raised decision slice may include:

```text
Reject
Defer
AcceptAlternative
NeedsManualCorrection
```

`AcceptAlternative` is valid only for detector-raised cases because alternatives are detector output.

Applying `AcceptAlternative` to a human-raised case is a typed error.

`NeedsManualCorrection` records that an issue appears real but the suggested alternative is absent, wrong, or insufficient.

`CustomReplacement` is deferred until replacement payload semantics are decided.

## HumanRaised and CustomReplacement

The accepted action for a human-raised case is closer to accepting a manual correction or custom replacement.

That requires replacement payload semantics:

* empty replacement
* insertion versus replacement
* newline handling
* whitespace handling
* cue splitting
* materialization constraints

Therefore, human-raised case acceptance and `CustomReplacement` belong to a later decision slice.

## Regression role

Detector-raised decisions mostly produce precision assets:

```text
the detector raised this
the human accepted / rejected / deferred / flagged it
```

Human-raised cases produce recall assets:

```text
the detector missed this
a human found it
future detectors should be evaluated against this class of miss
```

Human-raised cases may eventually become:

* missed-detection fixtures
* correction assets
* regression assets
* evidence for new `DetectionKind` categories

However, human-raised cases measure known misses, not true recall. Documentation must not overclaim full recall measurement from human-raised cases alone.

## Explicitly deferred

The following are deferred:

* `ReviewCaseId` generation scheme
* cross-run case identity
* applicability of cases and decisions under source revision change
* exact withdrawal event shape
* `CustomReplacement` payload semantics
* `AudioAnchor`
* insertion / zero-width anchors
* multi-anchor cases
* typed timing or segmentation findings
* `ManualFindingKind` taxonomy
* reviewer identity and attribution model
* persistence format
* materialized reviewed output

Applicability under source revision change is especially important and must be decided before persisted decisions are automatically reused across changed transcript revisions.

## Banned designs

The following designs are rejected:

1. Human findings bypass the review ledger.
2. Human findings are modeled as detector output under `AnalysisRun`.
3. Human testimony is stored inside `Evidence`.
4. `CorrectionDecision` is keyed directly to `CandidateKey`.
5. `DetectionKind::HumanReported` or any origin-as-detection-category.
6. Human-raised case creation automatically accepts a correction.
7. Mutable review or decision status is stored on `ReviewCase`.
8. `ReviewCaseId` is derived from `CandidateKey` in a way that makes detector identity the primary case identity and human-raised cases second-class.
9. Human-raised cases are deleted to correct mistakes.
10. Human-raised case creation events are omitted from persistence.
11. Case creation is modeled as a decision variant.
12. Human testimony is treated as recomputable evidence.

## Implementation consequences

The first decision implementation must introduce `ReviewCaseId` before recording decisions.

Even if the first implementation supports only detector-raised cases, decisions must bind to review case identity and observed transcript revision identity.

It is not acceptable to temporarily key decisions directly to `CandidateKey`.

The explicit `ReviewCaseOrigin` enum may wait until human-raised cases are implemented, but the ledger design must not assume detector-raised cases are the only possible origin.

This decision does not authorize persistence, custom replacement, reviewed-output materialization, audio support, speaker support, GUI work, or product CLI work.
