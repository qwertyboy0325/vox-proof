# 1. Executive judgment

This is a disciplined, well-governed early-stage repository whose engineering quality is ahead of its product evidence. The domain boundaries in code are genuinely good — better than most projects at this stage — and the documentation governance mostly does what it claims. But the project has three real problems, in descending order of severity:

1. **The product wedge is entirely unvalidated and the repo contains zero real data.** Every hypothesis in `docs/product/hypotheses.md` is honestly marked unvalidated, but the consequence isn't being acted on: the only detector is case-sensitive, byte-exact substring matching, which would produce near-zero useful findings on real ASR output of mixed Chinese-English speech. The implementation discipline is excellent; the product signal loop hasn't started.
2. **Revision identity is built on `DefaultHasher`, which is process-local.** This is acknowledged in docs as "not a stable cross-version fingerprint yet," but its full weight isn't stated anywhere: *nothing durable — decision records, regression assets, project bundles — can exist until this is replaced.* It's the single hardest gate in front of the entire flywheel vision, and it's currently listed as merely "open question 2."
3. **The docs and code disagree about normalization.** The architecture docs say analyzers work over a normalized view; the actual detector reads raw `Segment.text` and bypasses `normalized_view()` completely, which is currently a dead, clone-heavy identity copy.

Does the project have a real shot? Conditionally yes — but the shot is the *decision-record asset*, not the detector. Detection will always be beatable by bigger models. A local, deterministic, traceable, append-only record of human review decisions bound to content-addressed sources is something generic LLM tooling structurally does not produce. That is the defensible core, and it is also the part that doesn't exist yet.

# 2. Verified repository facts

All from direct inspection, not inference:

- **Code**: `src/lib.rs` (833 lines, ~570 of which are a single consolidated test module), `main.rs` (48-line CLI), `srt.rs`, `transcript.rs`, `anchor.rs`, `analysis.rs`, `candidate.rs`, `review.rs`, `pipeline.rs`. Zero dependencies (`Cargo.toml` has an empty `[dependencies]`), edition 2024.
- **Tests**: 56 unit tests in `lib.rs` plus 3 CLI integration tests in `tests/cli.rs` that spawn the binary with piped stdin. All were passing as of the last run this session.
- **Parser**: strict timestamp grammar (`HH{2,}:MM:SS,mmm`, minutes/seconds ranges enforced, checked arithmetic), parse errors distinct from validation issues, CRLF handled, empty cue text locked as `ValidationError::EmptyText`.
- **Anchoring**: `SourceAnchor` = revision + segment *position* (vec index, not SRT cue index) + half-open byte range, char-boundary-validated. `resolve()` refuses anchors from other revisions.
- **Revision identity**: `Transcript::revision_id()` hashes segments with `std::hash::DefaultHasher` (SipHash with unspecified stability), recomputed on every call, including once per `anchor()` call inside the detector loop.
- **Detector**: `detect_glossary_matches` does exact, case-sensitive `match_indices` per alias per segment; rejects duplicate aliases across entries as a config error; skips matches equal to the canonical term; clones the entire `GlossaryEntry` into `Evidence::Glossary` for every match.
- **Pipeline**: `run_glossary_review` composes run + detector + `ReviewCase::from`. Not wired to the CLI; `main.rs` only parses and prints validation issues.
- **Docs**: 8 documents plus `AGENTS.md`. `data-contract.md`'s status line was reconciled recently; the other seven docs still carry the older "Rust bootstrap exists; no end-to-end..." review line. `docs/vision/possibility-queue.md` exists and is not in the `docs/README.md` navigation. No CI configuration exists (no `.github/`). No fixtures, no sample SRT files, no license field.
- Both `pending-data-contracts.md` and `possibility-queue.md` refer to a "Material Decisions" / "material-decision process" — **no such document or process definition exists anywhere in the repository.**

# 3. Main criticisms

Ordered by how much they matter.

**C1 — The flywheel has no fuel and no fuel plan.** The vision is "human decisions become reusable assets." There is no human, no decision model, no real transcript, and no fixture in the repo. `docs/quality/evaluation.md` correctly says fixtures should be authorized short mixed zh-EN samples — but nothing has moved on this while five implementation slices shipped. The project is optimizing the machine before confirming the ore exists.

**C2 — `DefaultHasher` revision identity is a load-bearing placeholder.** `TranscriptRevisionId` values are not stable across Rust versions or necessarily across builds. Today nothing persists, so nothing breaks. But `CandidateKey`, `SourceAnchor`, and every future decision record embed this identity. The moment anything is written to disk, all of it becomes garbage on the next compiler upgrade. This must be promoted from "open question 2" to the explicit precondition for the entire persistence/decision/regression track, and it deserves a material decision (which also forces resolving whether the zero-dependency rule survives contact with the need for a real content hash — hand-rolled FNV-1a is defensible; hand-rolled SHA-256 is not a good use of time).

**C3 — Doc/code contradiction on normalization.** `architecture/overview.md`: "Transcript -> Normalize -> CandidateSpan detection...". Reality: `detect_glossary_matches` iterates `transcript.segments()` directly; `normalized_view()` is called by nothing except tests and does a full text clone to produce an identity copy. Either detectors should consume the normalized view (locking the coordinate-mapping discipline early, even while it's identity), or the view should be removed until non-identity normalization is actually decided. Right now it's a dead abstraction that contradicts the stated pipeline — exactly the kind of drift the project's own rules prohibit.

**C4 — The glossary detector's matching semantics are wrong for the stated wedge.** The wedge is mixed Chinese-English. That means: (a) Latin aliases need word-boundary semantics — alias `Kafka` currently matches inside `Kafkaesque`, and alias `RAG` matches inside `stoRAGe`-like tokens; (b) CJK text has no word boundaries, so substring matching is arguably correct there; (c) ASR casing is unreliable, so case-sensitive matching will miss most real occurrences. The doc comment honestly says case folding awaits the normalization gate — but that means the only detector is nearly inert on real data, and the normalization decision is now the bottleneck for any product value. No test currently locks the `Kafkaesque` false positive either, so the behavior is neither correct nor pinned.

**C5 — Evidence duplicates configuration, contradicting the contract's own principle.** `data-contract.md` says snapshot data "belongs at `AnalysisRun` scope and is not duplicated into every `CandidateSpan`." Yet `Evidence::Glossary` clones the full `GlossaryEntry` — canonical term plus *all* aliases — into every match. With a realistic language pack (pronunciation hints, per-entry metadata as envisioned), evidence becomes a per-match copy of config. The right long-term shape is a reference (pack snapshot identity + entry key) plus the matched form; the run snapshot owns the pack. Not urgent, but it's a live contradiction between contract language and code.

**C6 — Governance references a process that doesn't exist.** "Material Decisions" gates appear in two documents as the authorization mechanism for the most dangerous future work (speaker identity, persistence semantics). If the process isn't defined — even as one paragraph saying where such records live and what counts as one — the gate is decorative, and a future agent could reasonably conclude it's satisfied by nothing.

**C7 — Missing structural validation: overlapping and non-monotonic cue timings.** `validation_issues()` checks index consecutiveness, empty text, and per-segment duration, but two segments can overlap in time or appear in reverse chronological order with no issue raised. For a QA tool whose job is finding suspicious subtitles, this is a conspicuous gap, and it's cheap to add under the existing `ValidationError` model.

**C8 — Real-world SRT robustness gaps.** Verified against the parser: a UTF-8 BOM (`\u{FEFF}1\n...`) fails as `MalformedIndex` because `char::is_whitespace` doesn't strip U+FEFF — and BOMs are extremely common in SRT files from Windows tooling. Timing lines with trailing position metadata (`00:00:01,000 --> 00:00:02,000 X1:100`), which exist in the wild, are rejected by the strict millisecond check. Both may be *deliberate* strictness, but neither is documented as policy nor tested, so today they're accidents. Also: an empty input parses successfully as a zero-segment transcript with no validation issue — probably wrong for a QA tool, definitely undecided.

# 4. Product strategy review

**Is the wedge strong enough?** As a problem statement, yes: mixed zh-EN technical transcripts with names, acronyms, and product terms are genuinely painful, and "review with evidence, never silently rewrite" is a real differentiator against LLM-rewrite tools whose failure mode is confident meaning-alteration. As an executed wedge, not yet — see C1/C4. The wedge lives or dies on false-positive burden per accepted correction, and nothing measures that yet (`evaluation.md` names the metric; nothing feeds it).

**Is transcript QA the right wedge for the bigger vision?** Yes, and this is one of the strategically sound choices: transcript QA forces you to build exactly the primitives the vision needs (anchors, evidence, decisions, materialization) against the most tractable artifact (text). Audio-context detection bolted onto a weak review loop would be worthless; a strong review loop can absorb audio evidence later. The sequencing is right.

**The real moat, if this works:** the accumulated, traceable, revision-bound decision corpus plus organization-specific language packs — assets that are (a) local, so privacy-compatible; (b) deterministic and replayable, so regression-usable; (c) owned by the user. A frontier model can out-detect VoxProof any day; it cannot retroactively possess your organization's reviewed decision history. The moat is the ledger, not the detector. This should be stated this bluntly somewhere in the product docs; currently the north star ("audio-context-aware human review") emphasizes the more glamorous, more distant half.

**Who is the first serious user?** Unanswered anywhere in the repo, and it's the highest-value open product question. Best candidates: (1) Ezra, if there's a recurring personal transcript workload — dogfooding is the only realistic validation path for a solo project; (2) a zh-EN technical content team (podcast, courseware, dev-rel) producing subtitles weekly; (3) subtitle/localization vendors — though they're Windows-heavy, which matters for §8. If none of these can be named as a concrete person or team, the wedge is a hypothesis about strangers.

**What makes it irrelevant?** Upstream ASR getting good custom-vocabulary support (biasing lists, per-org fine-tunes) shrinks the error pool; LLM-in-the-editor "review my subtitles" flows get "good enough" for casual users. VoxProof survives in the segment where *auditability* is a requirement, not a preference. That's a narrower but real segment. Overclaim watch: nothing in the repo currently overclaims — the doc discipline has kept this honest, which is worth crediting.

**What can VoxProof do that generic tooling can't?** Deterministic reruns under pinned snapshots; decision records as first-class output; anchors that survive re-analysis; no-silent-rewrite guarantees you can actually verify (the materializer contract in `pending-data-contracts.md` §7 — explicit conflict reports instead of merged guesses — is the most differentiated single page in the repo).

# 5. Architecture / data model review

**What's right and should remain stable:** the five-way separation (finding / evidence / suggestion / review unit / future decision) is correctly modeled *in code*, not just prose — `CandidateAlternative` is genuinely non-binding, `ReviewCase` genuinely carries no status, detectors genuinely can't touch transcript state (no `&mut` path exists). `CandidateKey` as a transparent semantic struct rather than an opaque hash was the right correction. `SourceAnchor` binding revision into the anchor itself (rather than a later validity check) is the design decision that will save the project repeatedly. Keep all of it.

**Prematurely abstract:** only `NormalizedTranscript` (C3). It's small, but it's the one abstraction with zero consumers.

**Too concrete / future traps:**

- *Segment identity duality.* Anchors use vec position; `ValidationIssue` and `NormalizedSegment` use the SRT cue index (`Segment.index`), which is user data and can be duplicated (that's a validation issue, not a prevented state). Two indexing planes with similar names (`segment_position` vs `segment_index`) is a latent confusion trap; also, with duplicate cue indices, `ValidationIssue.segment_index` is ambiguous about *which* cue it refers to. Position should probably be the reporting coordinate everywhere, with cue index as display metadata. Worth a deliberate decision before more code accretes on both planes.
- *`anchor()` recomputing `revision_id()` per call* — already flagged as P3, still correct to defer, but note it compounds with C2: the fix (precomputed revision on an immutable `Transcript`) should land together with the stable-fingerprint change, not separately.
- *`candidate.rs` mixing finding types with detector implementation.* Fine at one detector; split at two. Don't pre-split.

**What should stay deliberately unresolved:** persistence layout, `CandidateRecordId`, cross-run decision carryover, `EditScript`, multi-anchor spans, ranking. The pending-contracts doc already holds these correctly.

# 6. Evidence / review / correction review

**Evidence model:** `Evidence` as a mandatory typed enum with one variant is the right starting point — narrow is correct here. The mistakes that would make the layer useless, in order: a `reason: String` variant (already banned in the contract, good); config duplication growing unchecked (C5); and evidence that can't resolve to source (currently impossible because anchors are validated at construction — keep that property). Long-term provenance shape: evidence carries *references into the run snapshot* plus observed facts (matched form, positions, scores); the snapshot owns configuration. Audio evidence later = a new `Evidence` variant referencing an `AudioAnchor` type that is *parallel to*, never a variant of, `SourceAnchor` — the two coordinate planes (parsed text bytes vs media time) must never share a type.

**Evidence vs inference vs decision:** the repo's implicit line is right: evidence = observed, reproducible facts under a snapshot; inference = derived judgments (future ranking scores — these belong in ranking output, not in `Evidence`); decision = human, append-only, canonical. The one place this will get tested is phonetic similarity: a similarity *score* is a computation, but it's reproducible under the snapshot, so it's legitimately evidence. Ranking's *weighting* of that score is not.

**`ReviewCase` and the path to `CorrectionDecision`:** the thin wrapper is correct and the restraint (no status field) is the single best example of discipline in the codebase. Before `CorrectionDecision` exists, three things must be true: (1) stable source fingerprint (C2 — non-negotiable); (2) decision ordering semantics decided (§4 of pending-contracts already sketches append-only events with monotonic per-case sequence — good sketch, promote it when ready); (3) a clear statement of what "applicable" means when a decision meets a changed source (open question 4). The **smallest safe slice** toward decisions: in-memory-only `CorrectionDecision` supporting `Reject` and `Defer` (no edit payload), with `ReviewStatus` derived from decision history. This proves the derived-status model and the append-only ordering with zero materialization risk and zero persistence dependency.

**Should rejected candidates become assets?** Yes, eventually, and this is under-emphasized in the flywheel framing: rejections are the training signal for the false-positive-burden metric that `evaluation.md` names as central. An asset model that only keeps acceptances measures recall and ignores the metric the product lives on.

**What reviewed output must prove:** byte-identical determinism (same source + same decisions → same output), and the conservative-failure property (any conflict → original preserved + explicit report, never partial output). The pending-contracts §7 rules are right; hold that line — "partial reviewed output mode" (open question 5) should stay unresolved as long as possible because the pressure to add it will come from convenience, not from the trust model.

# 7. Audio / speaker / regression-asset review

**Possibility queue framing:** genuinely good. The wedge/vision separation, the "nothing here authorizes implementation" rule, the Now/Later/Maybe/Dangerous taxonomy, and PQ-003/PQ-004 being marked Dangerous are all correct. Critiques of the framing rather than praise:

- The voice-cluster pipeline (`cluster → hypothesis → evidence → confirmation → confidence → scope → revocation`) is right but incomplete. Missing: **consent of the recorded person** (revocation-by-reviewer is not consent-by-subject), **retention limits**, **cluster merge/split semantics** (re-clustering changes cluster identity — the same identity-under-recomputation problem you already solved for transcripts, and it needs the same revision-bound treatment), and **jurisdiction** (voiceprints are regulated biometrics under GDPR and BIPA-style statutes; "local-first" reduces but does not eliminate exposure, especially for an org-wide deployment).
- The **minimal safe audio anchor** is missing from PQ-001's smallest slice in one respect: it needs a *stable audio content identity* (content hash of the media file) mirroring `TranscriptRevisionId`, or audio anchors inherit the same instability problem as C2. Time range + file path is not enough; paths move.
- **Hard technical problems** underrepresented in the queue: transcript-to-audio alignment when the SRT timings are themselves wrong (the timings are part of what's being QA'd — you cannot trust them as alignment ground truth), and audio evidence for *deleted/missing* speech, which has no text anchor at all and breaks the current one-anchor model. Neither needs solving now; both deserve a line so future-you doesn't design around their absence.
- **The mistake that permanently poisons the product:** storing voice embeddings linked to real names without subject consent and revocation, or training on review decisions without explicit opt-in. The second is already guarded in `evaluation.md` ("local processing does not imply training permission" — one of the best sentences in the repo). The first is only guarded by the possibility queue's "Dangerous" label; when audio becomes active, it needs a real material decision.

**Regression assets:** right flywheel. Minimal asset = fingerprinted source revision (or embedded excerpt), anchor, `CandidateKey`, evidence snapshot reference, decision, and expected outcome on rerun. Exclude initially: reviewer identity, rationale text (add rationale later — it's valuable but it's friction in the loop, and an empty-rationale corpus is worse than none), audio, and anything cross-run. **What makes the library compound:** every asset replayable under a pinned snapshot, so detector changes produce a diffable report. **What makes it junk:** assets bound to unstable hashes (C2 again — it is the root of everything), or assets stored without their snapshot so replays are meaningless. **Fixtures:** the first accepted correction should become a fixture the same week it exists. **Training data:** not until there's a corpus worth training on, an explicit opt-in, and a reason to believe a local model beats rules + glossary — likely never in v0.x, and that's fine.

# 8. Mac-first / cross-platform strategy review

**Current position (verified):** the repo is in the best possible platform position — zero dependencies, pure-Rust core, stdin/stdout CLI, nothing OS-coupled. Every platform option remains open. The strategic task is to *not spend this position* prematurely.

**Product/platform strategy.** Mac-first is a reasonable wedge for this product class: privacy-sensitive prosumers and content creators overindex on macOS, the workflow is keyboard-heavy desktop review, and audio work later benefits from a coherent media stack. The risks: notarization/signing friction for a solo dev shipping external binaries; and the possibility that the first *paying* segment (subtitle/localization vendors) is Windows-heavy. Mac-first becomes strategically wrong only if user discovery lands there — which is another reason the "first serious user" question precedes any platform commitment. Desktop-only first: yes, unambiguously. Mobile is at most a future capture/approve companion; full transcript QA on mobile is not a real workflow, and pretending it might be would distort the architecture. Don't design for it; just don't poison portability.

**Architecture boundary — what goes where:**

*Portable Rust core (everything currently in the repo, and everything on the current path):* domain model, parsing/validation, anchors, revision identity, detectors, evidence, review units, decisions, materialization, and — critically — the *serialization format of decision records and project bundles*. That format, not the code, is the true compatibility surface.

*Platform-specific (later, thin):* file pickers, audio capture/decode, waveform rendering, window/keyboard chrome, signing/notarization, model runtime binding.

*Must never couple to macOS:* anything inside domain records. The classic trap worth naming precisely: **macOS security-scoped bookmarks / file reference URLs inside persisted project data.** The moment a bundle stores those instead of relative paths + content hashes, the bundle is Mac-only forever. This rule costs nothing to adopt now and is expensive to retrofit.

*Abstractions worth creating early:* none as code. One as a rule: domain records contain no absolute paths, no OS-specific references, and identify external files by content hash with user-relinkable relative references. Document it; don't build a "storage abstraction layer" for it.

*Premature abstractions to refuse:* FFI bindings (UniFFI/swift-bridge), a command/IPC boundary, storage traits, audio adapters, model-runtime adapters. Each is a real cost with zero current consumers. The command boundary in particular should start as plain Rust function calls (which `run_glossary_review` already is) and only become serialized IPC when a process boundary actually exists.

**UI shell options, honestly:**

- **Rust core + SwiftUI/AppKit shell:** best keyboard UX, best native feel, best audio integration; cost is the FFI boundary and a second shell later for Windows/Linux. Trap: domain logic leaking into Swift view models — the exact "UI owns domain truth" violation the docs prohibit. Choose this only if Mac-native polish proves to be *the* differentiator with real users.
- **Tauri:** best fit for a solo dev wanting one shell across desktop; Rust-native so the core stays in-process; waveform via canvas is fine; sidecar binary bundling is supported. Traps: keyboard-heavy review UX in a webview takes deliberate work (focus management, IME behavior — which matters a lot for Chinese input); Linux webview inconsistency; and the gravitational pull toward reimplementing domain logic in TypeScript. Manageable traps.
- **Electron:** strictly dominated by Tauri here — heavier, worse memory story next to future local models, and the Rust core becomes a native-module/sidecar problem instead of in-process. No.
- **Flutter:** good canvas (waveform), viable Rust FFI, strong mobile path — but desktop text-editing and keyboard polish is historically its weak spot, and this product *is* keyboard-driven text review. Wrong center of gravity.
- **React Native/Expo:** mobile companion path only. Not a desktop review tool. Ignore until a companion app is real.
- **Kotlin Multiplatform / Compose:** JVM desktop + doubly-awkward Rust interop. Wrong ecosystem for a Rust-core project. No.
- **Web/PWA/local server:** a localhost server undermines the local-first trust story ("why is a server running?") and browser file access is friction. *But* — the cheapest first reviewer UI isn't a shell at all: a **static HTML review report** generated by the core (findings, evidence, context, anchors) is a zero-framework artifact that makes dogfooding possible immediately and defers the entire shell decision.

**Recommended platform strategy:**

- *Now, influencing core design:* the no-OS-paths-in-domain-records rule; treat decision-record and bundle formats as the portability surface; keep the core synchronous, in-process, and shell-agnostic (already true).
- *Document as future strategy:* Mac-first as a go-to-market hypothesis (not architecture), Tauri as default first shell, SwiftUI shell as the second option if native UX proves decisive, mobile as companion-only.
- *Defer:* the actual shell choice, FFI, persistence engine, audio adapters. Revisit trigger: the first slice that needs a human clicking accept/reject.
- *Must decide early (before persistence, not before UI):* stable content fingerprint (C2), decision-record serialization format, and project bundle layout (a directory bundle — `*.voxproof/` with a versioned manifest, content-addressed source copy, append-only `decisions.jsonl`, run snapshots — moves across OSes trivially precisely because the anchor coordinate plane is already parsed text, not file bytes; that earlier decision quietly bought bundle portability, and it's worth recognizing as such).
- *Would become expensive if ignored:* the fingerprint, the path rule, and bundle format versioning. Everything else is recoverable.

# 9. Testing / Rust / docs review

**Tests — good:** parser boundary coverage (grammar acceptance/rejection, parse-vs-validation separation), anchor error taxonomy, revision mismatch, detector config errors, canonical-form suppression, pipeline error propagation. The tests read as behavioral contracts, not line-coverage filler.

**Tests — missing, ordered by value:**

1. Overlapping / non-monotonic cue timings (requires the missing validation rule — C7).
2. BOM-prefixed input (decide the policy first; today it fails as `MalformedIndex`, which is almost certainly not the intended user experience).
3. Latin-substring false positive (`Kafkaesque` containing alias `Kafka`) — currently unpinned behavior in the core detector; either lock it as known-limitation or fix semantics first.
4. Empty input → zero-segment transcript (currently silently valid; probably should be a validation issue for a QA tool).
5. Overlapping matches from different aliases (e.g., `Apache Kafka` and `Kafka` as aliases of different entries) — the duplicate-alias guard doesn't cover overlap, and downstream ranking/materialization will need a defined behavior.
6. Whitespace-only cue text (the `trim()` path in `validation_issues` is live code with no direct test).

**Tests that risk freezing wrong behavior:** the CLI tests assert on human-readable substrings (`"parsed 1 segment"`), which is loose enough to be fine. The strict-timestamp rejection tests are the ones to watch — if the BOM/position-metadata policy decision goes toward leniency, several of them encode a stricter contract than the product may want. That's a policy question, not a test bug.

**Rust quality:** idiomatic and appropriately boring — no async, no trait objects, no premature generics, checked arithmetic in the timestamp math, correct `pub(crate)` discipline that makes `parse_srt` the only external door into `Transcript`. Two inconsistencies: `GlossaryEntry` exposes bare `pub` fields while everything else uses getters (pick one style); and the single 570-line test module in `lib.rs` should eventually migrate into per-module `#[cfg(test)]` blocks — do it when it hurts, not now. The `expect("a matched substring is always a valid char-boundary anchor")` in the detector is a genuinely sound invariant (match_indices guarantees boundaries), correctly documented. No rewrites justified.

**Docs:** unusually high quality, at the edge of proportionality for one developer. Concrete defects: the phantom Material Decisions process (C6); seven of eight docs carrying stale "Last reviewed against code" lines while `data-contract.md` was updated — the per-doc review line is becoming a maintenance tax that will drift chronically, and it might be worth consolidating "implementation status" into exactly one place; `possibility-queue.md` absent from navigation (an "Exploratory" section in `docs/README.md` keeps the index truthful without canonizing it); and the normalization contradiction (C3). The doc most likely to mislead a future agent is `pending-data-contracts.md` section 4 — its `SourceSelection` design is detailed enough to look implementable; its disclaimers are adequate, but only if agents read headers. The possibility queue's anti-drift rules are, by contrast, exemplary.

**One process gap:** no CI. The project's discipline currently lives in session habits (`cargo fmt/clippy/test` before commits). A minimal GitHub Actions workflow running exactly those three commands would make the discipline structural instead of behavioral. It's the cheapest governance upgrade available.

# 11. Open questions

1. Who is the first user, and does Ezra personally have a recurring transcript workload to dogfood against? (Determines everything downstream, including Mac-first.)
2. Stable fingerprint algorithm — and does the zero-dependency rule bend for it?
3. SRT leniency policy: BOM, trailing timing metadata, empty files — strict parser with documented rejections, or tolerant parser with validation issues?
4. Matching semantics for the wedge languages: where exactly does word-boundary logic live once Latin and CJK need different rules — detector, normalization view, or language pack metadata?
5. What is a Material Decision, concretely, in this repo?
6. When decision records exist, what does "applicable" mean for a decision whose source fingerprint no longer matches (pending-contracts open question 4) — and who decides it, given it's the semantic heart of the trust model?
7. Is the reporting coordinate for humans the cue index or the segment position, given cue indices can be duplicated?

The short version: the machine is well-built and honestly documented; the two things standing between it and a real product are a stable identity for what it reviews, and a single real transcript to review. Both are addressable this month.

# 12. Fourth-round discussion: human-raised review cases

This follow-up discussion challenged a major workflow assumption in the current model: the visible review flow is detector-driven.

```text
detector
-> CandidateSpan
-> ReviewCase
-> human decision
-> future regression asset
```

That flow handles findings the system already detects. It does not, by itself, explain what happens when a human reviewer notices a wrong term, missing phrase, wrong name, bad segmentation, timing issue, or other transcript problem that no detector raised. If the long-term value of VoxProof is the human-reviewed decision ledger and regression asset, missed detections must also become first-class assets. Otherwise the system can only learn from what it already detects, and recall failures never enter the loop.

## 12.1 Main conclusion

This is a real product gap, but not an architectural emergency.

The product gap is serious because a detector-only ledger measures "was the detector right when it spoke" but not "what did the detector miss." Human-raised cases are the natural mechanism for recording recall failures, seeding future detector categories, and grounding future `DetectionKind` growth in real uncovered review cases rather than speculative taxonomy design.

The architecture is not yet boxed in because `ReviewCase` is already distinct from `CandidateSpan`. That earlier restraint matters: the system can later make `ReviewCase` origin-aware without treating `CandidateSpan` as the only possible review subject.

## 12.2 Conceptual model discussed

The preferred conceptual direction is:

```text
ReviewCase
  origin:
    DetectorRaised(CandidateSpan)
    HumanRaised(ManualFinding)

ManualFinding
  SourceAnchor
  human observation
  optional note
  optional proposed correction
```

The key semantic distinction:

```text
CandidateSpan = deterministic derivation from an AnalysisRun
ManualFinding = human-created event scoped to a transcript revision
```

A detector-raised case is run-scoped and should be reproducible from the same analysis snapshot. A human-raised case is not reproducible by rerunning detectors. It is a reviewer event bound to a source revision. Therefore, modeling humans as another detector would be misleading and would weaken the meaning of `AnalysisRun`.

The center of the model should not be `CandidateSpan`. The center should be:

```text
SourceAnchor
+ ReviewCase
+ decision ledger
```

`CandidateSpan` is one origin feeding that review system, not the whole review system.

## 12.3 Evidence, observation, and decision boundaries

The discussion separated three concepts that should not be collapsed:

- `Evidence`: machine-derived, typed, reproducible facts under an analysis snapshot.
- Human observation: testimony or reviewer-authored finding content.
- Decision: canonical human action against a review case.

The conclusion was that `Evidence::HumanObservation` would be the wrong shape. A human observation is not machine-recomputable evidence, and it is not itself a decision. It is the content of a human-raised finding. Raising a case and deciding a case must remain separate events, even if the same person performs both actions immediately.

This preserves the no-silent-rewrite rule:

```text
human raises issue
-> ReviewCase exists
-> human decision is recorded separately
-> only then can correction/materialization logic eventually act
```

## 12.4 Anchor boundaries

For a first text-only version, a human-raised case should require a `SourceAnchor`. That keeps it revision-bound and keeps the current source-preserving contract intact.

Important boundary distinctions:

- Text-only term/name/phrase correction: can usually be anchored to existing transcript text.
- Semantic knowledge issue: may start as anchored observation plus note, but should not prematurely become a detector taxonomy.
- Timing or segmentation issue: text anchors are imperfect; typed timing/segmentation anchors should wait for a deliberate model.
- Missing speech with no transcript span: should wait for an `AudioAnchor` or a future insertion/zero-width anchor model.
- Audio-backed issue: should wait until audio identity and audio anchors exist.

For missing phrase cases before insertion anchors exist, the safest interim interpretation is to anchor nearby text and record the omission in the note. That preserves the signal without pretending the current `SourceAnchor` model can express pure insertion or audio-only evidence.

## 12.5 Effect on regression assets

Human-raised cases should eventually feed the regression loop in multiple ways:

- as missed-detection fixtures, testing whether future detectors now catch the issue;
- as correction assets, once an accepted correction exists;
- as evidence for new detector kinds or matching semantics;
- as false-negative examples for evaluating recall.

The primary role is missed-detection signal. Accepted corrections are valuable, but if the ledger only preserves acceptances from detector-raised cases, it still cannot measure what detectors failed to raise.

## 12.6 Relationship to the first decision slice

The discussion identified a category error in treating `CreateManualReviewCase` as a decision variant.

`CreateManualReviewCase` is not a decision against a case. It is a way for a case to come into existence. Decisions should remain actions applied to existing review cases.

Therefore, the first decision slice can still stay narrow:

```text
Reject
Defer
AcceptAlternative
NeedsManualCorrection
```

But the decision-record design should assume that review cases can eventually have multiple origins. The important early rule is:

```text
Decision records bind to review-case identity and observed source fingerprint,
not directly to CandidateKey or detector provenance.
```

That rule prevents future human-raised cases from requiring a disruptive schema change. `AcceptAlternative` is meaningful only for detector-raised cases because alternatives are detector output. The equivalent action for a human-raised case is closer to accepting a proposed custom replacement, which likely belongs with the later `CustomReplacement` / manual correction payload design.

## 12.7 What should be decided now

The minimal non-implementation decision to carry forward:

1. `ReviewCase` should not be permanently defined as detector-only.
2. Future decision records should bind to review cases, not to `CandidateKey`.
3. Human-raised findings are revision-scoped events, not analysis-run outputs.
4. Raising a finding and deciding a finding are separate events.
5. `AcceptAlternative` is detector-origin-specific; human-origin acceptance needs a custom/manual correction payload.

This likely belongs in or near the future Material Decision about decision records and review-case identity.

## 12.8 What should be deferred

The following should not be implemented yet:

- `ManualFinding` code;
- manual case creation UI;
- persistence for manual findings;
- audio anchors;
- zero-width/insertion anchors;
- timing/segmentation-specific finding types;
- a broad `ManualFindingKind` taxonomy;
- free-form materialization or editing.

The safe deferral argument is that the first private fixture and manual correction pass can record missed issues outside the system for now. Those notes can become the seed material for the later human-raised case design.

## 12.9 Smallest safe future slice

When the time comes, the smallest safe implementation slice would be:

```text
ReviewCaseOrigin
  DetectorRaised(CandidateSpan)
  HumanRaised(ManualFinding)

ManualFinding
  SourceAnchor
  note
  proposed_correction: Option<String>
```

The slice should validate that the anchor belongs to the current transcript revision, allow shared decision handling for origins where the decision is meaningful, and reject `AcceptAlternative` for human-raised cases with a typed error. It should not include audio, persistence, UI, ranking, materialization, or a full taxonomy.

## 12.10 Updated sequencing implication

The converged sequence does not need a major reorder. It needs one explicit addition:

```text
acquire private real mixed zh-EN fixture
add minimal repo-level Material Decision process
define stable TranscriptRevisionId
add eval harness
let real fixture data inform matching semantics
later add report artifacts
later add first detector-raised decision slice
later add human-raised cases + custom replacement/manual correction payload
```

The human-raised case concept should influence the decision-record Material Decision before implementation. The code implementation can wait until the detector-raised decision path proves the ledger mechanics.

## 12.11 Dangerous wrong designs

The most dangerous wrong designs discussed:

1. Letting human-raised findings bypass the decision ledger because "a human already said so."
2. Treating a human as just another detector and injecting manual findings into `CandidateSpan` / `AnalysisRun`.
3. Adding `Evidence::HumanObservation` and mixing testimony with recomputable machine evidence.
4. Keying durable decisions directly on `CandidateKey`, which would make non-detector origins hard to support later.
5. Adding a catch-all `DetectionKind::HumanReported`, which would pollute the detector taxonomy with workflow origin.
6. Building audio, timing, insertion, persistence, or UI machinery before the text-only decision ledger is proven.

The short conclusion: human-raised review cases are real and strategically important, but the safe move now is to preserve the conceptual opening in the decision-record design, not to implement the feature immediately.
